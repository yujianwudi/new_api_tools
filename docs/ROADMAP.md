# NewAPI-Tool 产品与技术路线图

> 本路线图面向使用 [QuantumNous/new-api](https://github.com/QuantumNous/new-api) 运营 API 中转站的团队。推进顺序是：**先保证可用和安全，再建立可观测性，随后治理利润与成本，最后引入可解释风控和平台化能力**。

## 1. 产品定位

NewAPI-Tool 是 NewAPI 的**旁路运营与管理控制台**，不是新的 API 网关，也不是 NewAPI 的替代实现。它应帮助运营者回答五类问题：

1. 系统现在是否安全、可用，数据是否可信？
2. 哪个渠道、模型或任务正在影响稳定性？
3. 收入、成本、毛利和额度消耗分别来自哪里？
4. 哪些用户或行为值得复核，证据是什么？
5. 多个 NewAPI 实例能否用统一方式治理？

路线图采用“阶段门槛”而不是固定日期。只有上一阶段的验收指标持续达标，下一阶段才进入默认开启状态。

## 2. 不可破坏的架构边界

以下约束适用于所有阶段，优先级高于单个功能需求：

| 边界 | 约束 |
|---|---|
| 旁路接入 | NewAPI-Tool 以独立进程或容器运行，只连接 NewAPI 已有数据库、缓存或受支持的接口；**不代理模型请求，不进入 NewAPI 数据面主链路**。 |
| 不修改上游 schema | **不得在 NewAPI 数据库中执行 `CREATE/ALTER/DROP TABLE`、新增索引、触发器或其他迁移。**需要持久化的新配置、任务、快照、风控案件和审计数据必须写入 NewAPI-Tool 自有数据库。 |
| 读优先、写受控 | 默认能力以查询、分析和预览为主。确需修改 NewAPI 现有数据时，只使用上游已经存在的字段和语义，并提供权限校验、影响范围预览、事务、幂等保护和审计记录。 |
| 故障隔离 | NewAPI-Tool 停止、升级、数据库不可用或缓存失效时，不得影响 NewAPI 的模型请求、计费和用户登录。 |
| 数据不足时安全降级 | 数据源未知、日志不完整、缓存过旧或查询超时时，页面应明确标记“不完整/已降级”；删除、封禁等高风险写操作应拒绝执行，而不是把未知当作零。 |
| 兼容而非耦合 | 通过版本识别、能力探测和适配层兼容 NewAPI 版本差异；不得把某次生产库导出当作永久不变的接口契约。 |
| 指标可追溯 | 每个运营指标都应能说明数据源、过滤条件、时间范围、时区、币种、计算公式和更新时间。 |

## 3. 阶段总览

| 阶段 | 主题 | 主要结果 | 状态 |
|---|---|---|---|
| P0 | 可用性与生产安全 | 可以安全部署、登录、读取和执行受控管理操作 | 已发布 v0.2.0 |
| P1 | 稳定性与可观测性 | 能快速发现、定位和恢复控制台及数据任务故障 | 下一阶段 |
| P2 | 盈利与成本治理 | 能解释收入、消耗、成本和毛利，并给出可验证建议 | 待启动 |
| P3 | 可解释风控 | 风险告警有证据、可复核、可回滚，不依赖黑盒自动处罚 | 待启动 |
| P4 | 平台化 | 支持多实例、多角色、扩展接口和高可用运营 | 待启动 |

## P0：可用性与生产安全（已发布 v0.2.0）

### 目标

建立生产环境最低安全基线：数据含义正确、危险操作不会因日志缺失而误伤、认证和外部请求边界明确、默认部署不暴露不必要的攻击面，并形成可重复的构建与测试流程。

### 核心交付

#### 数据正确性与破坏性操作保护

- 显式区分主日志库、独立日志库、回退日志源和未知状态。
- 独立日志库失败时允许只读回退，但删除、清理等依赖完整日志的操作必须安全失败。
- 批量删除前检查日志源健康度、数据新鲜度、空日志和未来时间戳。
- “从未活跃”等判断基于完整计费日志历史，避免把日志不可用误判为无活动。
- 用户硬删除、清理、批量删除、封禁和解封使用数据库事务；任一步失败时整体回滚。
- NewAPI Redis 未被明确确认关闭时，所有用户、Token、分组和 IP 设置的直接写库操作安全失败；永久删除兼容路径默认关闭，并优先迁移到 NewAPI 官方管理 API。
- Token 更新错误必须向上返回；封禁/解封不能错误复活已过期或已耗尽的 Token。
- 修正风险阈值、签到额度字段、同 IP 注册口径、兑换码状态、Token 状态和额度换算等已知数据语义问题。
- 额度换算拒绝 `NaN`、无穷值和整数溢出；随机额度使用密码学安全随机源。

#### 安全边界

- JWT 只接受配置的精确签名算法，不接受算法降级或替换。
- API Key、Token 和其他密钥在日志、错误和页面中默认脱敏。
- CORS 使用显式允许列表；登录端点具备限速和失败保护。
- 客户端 IP 只在直接对端命中精确可信代理列表时解析 XFF，并从右向左剥离可信代理，公网直连不能通过伪造转发头绕过限流。
- 所有可配置外部 URL 防护 SSRF：阻止环回、私网、链路本地和危险重定向。
- Abuse Hub 等携带凭证的外部连接仅允许 HTTPS，并限制响应体大小和读取时间。
- 公共模型状态接口限制请求频率、批量规模、请求体和时间窗口。

#### 部署与供应链

- 对外服务默认绑定 `127.0.0.1`，由用户显式决定是否暴露到公网。
- 生产镜像固定可信基础镜像版本，并以非 root 用户运行。
- Nginx 使用明确的 `/api/` 路由优先级，避免静态资源回退吞掉 API 请求。
- 更新存在已知高危漏洞的生产依赖，补齐前端 lint 配置。
- CI 至少执行 Go 测试、前端 lint、前端构建和依赖安全审计。
- 安装脚本、Compose 配置和环境变量示例保持相同的安全默认值。

### 验收指标

- `go test ./...`、前端 lint 和生产构建全部通过；核心安全回归测试进入 CI。
- 已知 Critical 级安全问题为 0；无法立即修复的 High 级问题必须有风险说明、负责人和截止版本。
- 所有删除、封禁、解封和批处理写操作都有事务回滚或失败注入测试。
- 在日志库不可用、日志过旧或日志为空的测试中，高风险批量操作 100% 被阻止，并返回可理解原因。
- JWT 算法替换、登录爆破、CORS 越权、私网 SSRF、重定向绕过和超大响应均有自动化负向测试。
- 全新环境可以按文档启动并通过健康检查；默认部署不直接监听所有网卡。
- 停止 NewAPI-Tool 后，NewAPI 模型请求链路和计费链路保持不受影响。
- P0 发布必须提供回滚步骤，目标是在 15 分钟内恢复到上一可用镜像。

### 边界 / 本阶段不做

- 不建设多节点高可用、跨区域容灾或多租户。
- 不引入自动调价、自动切换渠道或自动封禁。
- 不修改 NewAPI 表结构，也不以“性能优化”为由向上游数据库添加索引。
- 不承诺完整财务核算或风险识别准确率；P0 只保证现有数据不被明显误读和误写。
- 不重做现有 UI 信息架构，除非界面会导致危险误操作或无法表达降级状态。

## P1：稳定性与可观测性

### 目标

让运营者在故障影响扩大前发现异常，并能区分是 NewAPI、主数据库、独立日志库、Redis、外部服务还是 NewAPI-Tool 自身的问题；同时保证大表查询和后台任务不会拖垮上游生产库。

### 核心交付

#### 可观测性基线

- 提供结构化日志，统一请求 ID、任务 ID、数据源、耗时、结果和错误分类。
- 暴露 Prometheus/OpenTelemetry 兼容指标：请求量、错误率、延迟、数据库连接池、查询超时、缓存命中率、任务积压和数据新鲜度。
- 区分存活检查、就绪检查和依赖健康检查；健康接口不得执行昂贵全表查询。
- 建立默认 SLO 看板和告警规则，覆盖控制台 API、数据同步任务、日志新鲜度和外部依赖。
- 对页面展示“数据时间”和“降级原因”，避免把缓存结果伪装成实时结果。

#### 查询与任务韧性

- 所有大表查询具备超时、分页、最大时间窗口和扫描预算。
- 重型聚合改为可取消的后台任务，支持进度、重试、指数退避和失败归档。
- 任务具备幂等键和并发锁，防止多副本或重复点击造成重复写入。
- 缓存采用一致的键版本、TTL、随机抖动和击穿保护；缓存不可用时限制回源并发。
- 对主库和日志库分别设置连接池、只读查询约束和熔断策略。

#### 兼容性与运维

- 建立 PostgreSQL、MySQL，以及主日志表/独立日志库两种部署形态的兼容测试矩阵。
- 建立版本化 NewAPI 管理 API 适配层，优先迁移封禁/解封、用户分组和活动用户永久删除；管理员 Token、全站 Setting、软删除与批量 purge 在上游缺少受支持接口时继续安全失败。
- 启动时探测 NewAPI 版本和可用能力；不兼容字段应禁用相关功能并明确提示。
- 为安装、升级、回滚、日志库切换、缓存清理和自有数据库恢复编写 Runbook。
- 对 NewAPI-Tool 自有状态做备份和恢复演练；不接管 NewAPI 数据库备份职责。
- 建立固定规模的查询基准数据集和性能回归测试，版本间可比较。

### 验收指标

- 进入稳定发布后，NewAPI-Tool 控制面 30 天可用性目标不低于 99.9%，且不把上游计划维护计入自身可用性。
- 标准基准数据集（至少 1,000 万条日志、10 万用户、30 天窗口）下，常用只读接口 p95 小于 2 秒；重型分析在 60 秒内异步完成或返回明确进度。
- 控制台 API 5xx 比例低于 0.5%；所有超时均有可检索的错误分类和请求 ID。
- 日志停止更新后 5 分钟内产生“数据过旧”告警，页面同时显示最后成功时间。
- Redis、独立日志库或任一外部服务故障演练中，不产生无限重试、请求雪崩或非预期写操作。
- PostgreSQL/MySQL 与日志分库/不分库的核心契约测试全部通过。
- 每个告警都有负责人、处置步骤和恢复判断；每季度至少完成一次自有状态恢复演练。

### 边界 / 本阶段不做

- 不自动执行会影响价格、渠道、额度或用户状态的业务决策。
- 不要求用户绑定单一监控厂商；指标和日志优先采用开放格式。
- 不直接修改 NewAPI 数据库索引；若发现慢查询，只输出证据和上游优化建议，或改用 NewAPI-Tool 自有快照/聚合存储。
- 不为了“实时”而持续扫描全量日志，允许对重型分析采用有明确时间戳的近实时结果。

## P2：盈利与成本治理

### 目标

把“额度很多、请求很多”转化为可经营的数据：明确现金流入、已确认消耗收入、供应商成本、毛利和毛利率，并定位亏损模型、渠道、用户组与异常消耗。

### 核心交付

#### 统一经营口径

- 建立版本化指标字典，分开定义充值现金流入、赠送额度、已消耗额度、确认收入、渠道成本、毛利和毛利率。
- 明确成功计费请求的过滤条件，统一模型倍率、分组倍率、缓存计费、补偿和退款的处理方式。
- 所有金额保留原币种、汇率来源、换算时间和舍入规则，避免把不同币种直接相加。
- 在 NewAPI-Tool 自有数据库保存渠道价格、合同阶梯价、汇率和规则快照；历史报表使用当时版本，不被新配置重算覆盖。

#### 成本归因与利润看板

- 按渠道、模型、用户、用户组、Token 和时间维度归因请求量、Token、收入、成本和毛利。
- 展示“无法归因成本”和“估算成本”的占比，估算值不得与已确认成本混为一谈。
- 提供亏损模型、低毛利用户、渠道价格漂移、异常赠送额度和高失败成本排行榜。
- 支持按日对账和差异下钻，从汇总指标追溯到原始日志或规则快照。

#### 预算、预测和建议

- 为渠道、模型和用户组设置成本预算与毛利护栏，支持日/周/月消耗速度告警。
- 基于历史趋势提供余额耗尽时间、月末成本和毛利区间预测，并展示置信范围。
- 对调价、倍率调整和渠道切换提供“假设模拟”，输出预计收益、风险和受影响范围。
- 提供 CSV/API 导出，便于运营、财务和采购复核。

### 验收指标

- 每日成功计费日志与 NewAPI 对应消耗口径的可归因率不低于 99.5%；剩余差异可按原因分类。
- 已确认渠道价格覆盖的成本占比不低于 98%；未覆盖部分在所有利润页面显著标记。
- 100% 的利润指标可查看公式、数据源、时间范围、时区、币种和规则版本。
- 日级汇总与抽样原始日志重算差异低于 0.5%，并有自动化对账测试。
- 成本或毛利越过配置护栏后 5 分钟内产生告警；重复告警具备聚合和恢复通知。
- 所有调价、倍率和渠道建议先提供模拟结果，默认不直接写入 NewAPI。

### 边界 / 本阶段不做

- 不替代会计、支付网关、税务或银行对账系统，不把充值记录直接等同于会计收入。
- 不在缺少供应商价格时伪造“精确毛利”，只能显示未知或有标签的估算区间。
- 不自动调价、不自动改倍率、不自动切换渠道；业务变更仍需管理员确认。
- 不在 NewAPI 表中增加成本列或快照表，所有经营规则和历史快照保存在工具自有数据库。
- 不以毛利优化为由牺牲 P1 已定义的可用性和延迟护栏。

## P3：可解释风控

### 目标

建立“发现—取证—复核—处置—复盘”的闭环。风险判断必须说明触发原因和证据，运营人员可以确认、驳回或回滚，避免不透明模型直接伤害正常用户。

### 核心交付

#### 证据与规则

- 建立统一风险事件模型，记录主体、行为、时间窗口、数据源、证据完整度和关联身份。
- 支持同 IP、Token 轮换、异常频率、额度突增、失败率、共享身份和 Abuse Hub 通报等可解释规则。
- 规则具备版本、阈值、适用范围、启停状态、负责人和变更记录。
- 构建轻量身份关联图，但对 email、OAuth ID、IP 等敏感字段采用最小化展示、哈希或脱敏策略。
- AI 仅用于摘要、聚类或辅助排序；原始证据和确定性规则始终可查看。

#### 案件与人工复核

- 风险告警进入案件队列，支持分派、备注、证据快照、相似案件和处理时限。
- 提供确认风险、误报、信息不足等标签，形成后续阈值评估所需的标注集。
- 封禁、限额、Token 禁用等动作必须展示影响预览、理由和有效期。
- 高风险写操作使用幂等键、事务和完整审计，能恢复的动作必须提供回滚入口。
- Abuse Hub 数据记录来源节点、签名/认证结果、拉取时间和可信等级；外部通报不能直接等同于本地定罪。

#### 影子模式与效果评估

- 新规则先在影子模式运行，只告警、不处置，并比较命中率、误报率和运营工作量。
- 通过已标注案件评估准确率、召回率、误报率和不同用户组之间的偏差。
- 规则阈值变化提供前后回放，展示过去一段时间会新增或减少哪些告警。
- 对规则漂移、数据中断和证据不完整建立监控。

### 验收指标

- 100% 的风险告警包含原因码、规则版本、证据来源、时间窗口和数据新鲜度。
- 100% 的处置动作记录操作者、目标、原因、前后状态、请求 ID 和结果；重复提交不会重复生效。
- 新规则默认完成不少于 14 天且至少 500 个已复核样本的影子评估；样本不足时保持“实验”状态。
- 每条正式规则都有基线误报率、目标区间和停用阈值；超过阈值自动停止建议升级，只保留告警。
- 可回滚动作的恢复演练成功率为 100%；不可逆动作必须二次确认并限制批量范围。
- 外部通报、AI 建议或单一弱信号均不能单独触发永久封禁。

### 边界 / 本阶段不做

- 不允许黑盒 AI 直接封禁用户，也不以单个 IP 或单次失败请求作为永久处罚依据。
- 不在不同运营站点之间自动共享明文身份、密钥或完整请求内容。
- 不把相关性描述成因果关系，不把“高风险”描述成“已违规”。
- 不建设绕过 NewAPI 权限模型的隐藏管理通道。
- 不追求零风险；优先保证证据质量、人工复核效率和误伤可恢复性。

## P4：平台化

### 目标

在单实例运营能力成熟后，将 NewAPI-Tool 演进为可管理多个 NewAPI 实例、多个团队和扩展模块的运营平台，同时保持旁路隔离和兼容性。

### 核心交付

#### 多实例与权限

- 一个控制面管理多个 NewAPI 实例，每个实例拥有独立连接、密钥、指标、任务和策略范围。
- 提供组织、项目、环境和实例层级的 RBAC，支持最小权限、临时授权和审批流。
- 支持 OIDC/SSO、服务账号、密钥轮换和完整登录审计。
- 所有缓存键、任务锁、导出文件和自有数据库记录包含明确租户/实例边界。

#### 适配与扩展

- 建立版本化 NewAPI 适配器接口，通过能力清单决定页面和任务是否可用。
- 提供只读优先的开放 API、Webhook 和事件订阅，用于告警、报表和工单集成。
- 插件使用权限声明和隔离执行环境，默认不能访问原始数据库或平台密钥。
- 支持配置即代码、策略导入导出、差异预览和分环境发布。

#### 高可用与规模化

- 后台任务支持多副本选主、分片、幂等消费和失败队列。
- 控制面支持无状态横向扩容，自有数据库支持备份、时间点恢复和滚动升级。
- 建立租户级限流、配额和资源成本统计，防止单个实例拖慢整个平台。
- 提供审计导出、数据保留策略、删除流程和合规配置。

### 验收指标

- 租户与实例隔离测试覆盖 API、缓存、任务、导出和审计数据，跨租户越权用例 100% 被拒绝。
- 关键权限变更、密钥操作和批量写操作 100% 进入不可抵赖审计链。
- 标准平台基准（20 个 NewAPI 实例、100 个并发管理员）下，常用接口 p95 小于 2 秒，后台任务不会互相饿死。
- 控制面 30 天可用性目标不低于 99.9%；单副本故障不影响任务最终执行，也不影响任一 NewAPI 数据面。
- 支持的 NewAPI 版本均通过适配器契约测试；遇到未知版本时默认禁用不安全写能力并给出升级建议。
- NewAPI-Tool 自有状态恢复目标为 RPO 不高于 5 分钟、RTO 不高于 30 分钟；恢复演练至少每季度一次。
- 插件越权、资源耗尽和密钥读取的负向测试全部通过。

### 边界 / 本阶段不做

- 不把 NewAPI-Tool 演变为模型请求代理或重新实现 NewAPI 数据面。
- 不维护侵入式 NewAPI fork，不要求上游接受 NewAPI-Tool 专用 schema。
- 不在 P0—P3 的单实例价值和指标尚未稳定前提前建设复杂多租户架构。
- 不成为通用 CRM、支付、客服或财务平台；通过 API/Webhook 与专业系统集成。
- 不允许插件绕过平台权限直接修改 NewAPI schema 或读取其他租户数据。

## 4. 交付方式

### PR 拆分原则

- 每个 PR 只解决一个可验证问题，附根因、影响范围、回滚方式和测试证据。
- 风险较高的能力先以 feature flag、只读模式或影子模式交付。
- NewAPI-Tool 自有数据库迁移必须向前兼容，并提供备份与回滚说明；不得把迁移脚本指向 NewAPI 数据库。
- 涉及 PostgreSQL/MySQL 差异、日志分库或 NewAPI 版本差异时，PR 必须补充对应契约测试。
- 任何批量写功能都必须同时交付预览、上限、权限、事务、审计和失败处理。

### 建议的后续 PR 顺序

1. **P0（v0.2.0 已发布）：生产安全基线**——完成数据安全、认证边界、SSRF 防护、安全部署和 CI。
2. **P1-1：观测基础**——结构化日志、指标、健康状态和数据新鲜度。
3. **P1-2：查询与任务韧性**——查询预算、异步任务、幂等、缓存击穿保护。
4. **P1-3：兼容与运维**——数据库矩阵、版本能力探测、基准测试和 Runbook。
5. **P2-1：指标字典与价格快照**——先统一经营口径，再制作利润页面。
6. **P2-2：归因与对账**——成本覆盖率、差异下钻和经营告警。
7. **P3-1：风险事件与案件队列**——先完成证据闭环，再讨论自动化。
8. **P3-2：影子规则与效果评估**——用标注数据决定阈值和上线范围。
9. **P4：多实例基础**——仅在单实例稳定性、利润指标和风控流程均达到阶段门槛后启动。

## 5. 每阶段统一完成定义

一个阶段只有同时满足以下条件才算完成：

- 产品行为、失败行为和数据口径有文档，页面能表达未知、估算、过期和降级状态。
- 单元测试、集成测试、负向安全测试和必要的性能回归测试进入 CI。
- 没有未处理的 Critical 问题；High 问题有明确处置计划且不突破本路线图边界。
- 安装、升级、回滚和故障处置可以按 Runbook 复现。
- 对 NewAPI 数据面的隔离经过验证，关闭 NewAPI-Tool 不影响上游核心服务。
- 所有新增持久化数据位于 NewAPI-Tool 自有存储，未修改 NewAPI schema。
- 指标达到验收门槛并持续一个完整观察窗口，而不是只在单次演示中通过。
